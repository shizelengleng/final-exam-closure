### 第二轮：5 项需求实现（本次会话）

#### 需求 F: 搜索引擎重构

**问题**: 搜索结果质量差，同一 URL 重复出现，中国站点大面积超时，DDG 中文搜索失败

**参考项目**: 从 GitHub 分析了 4 个搜索聚合项目：
- `chen150450/multi-search-aggregator` — 30+ 引擎并行 + 去重 + 多信号排序
- `erguotou520/aggregated-search` — 优先级降级链 + `Promise.allSettled` 兜底
- `Jesseovo/PaperCash` — 学术加权评分（相关性 40% + 引用 25% + 新鲜度 15% + 权威度 15%）
- `puzzithinker/zlib-searcher` — 多字段查询解析

**核心改进**:

1. **URL 标准化去重** — `normalizeUrl()` 去掉追踪参数（`utm_*`、`ref` 等）、`www.` 前缀、统一路径格式
2. **来源权威度权重** — 每个搜索引擎 0.35~0.95 权重分（Semantic Scholar 0.95，pansou 0.35）
3. **跨引擎交叉验证** — 同一 URL 被多个引擎找到时 +0.08/源，上限 +0.25
4. **`Promise.allSettled`** — 某引擎崩溃不拖垮整个搜索
5. **全局 45 秒超时** — 搜索不会无限卡死
6. **DDG HTML 兜底** — API 失败时用 BrowserWindow 抓 DuckDuckGo HTML
7. **反爬检测** — 识别百度/Bing 验证码页面，返回空结果而非垃圾
8. **AnySearch API 集成** — JSON-RPC 2.0 调用 `api.anysearch.com`，支持 23 个垂直领域
9. **AnySearch 内容提取** — `fetchPageAsMarkdown` 优先用 AnySearch API 提取，失败再用 BrowserWindow

**文件**: `electron/services/searchEngine.ts`, `electron/services/browserSearch.ts`, `electron/services/anysearch.ts`（新建）, `config/searchSources.json`

## 架构决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 文件查看器 | `@cyntler/react-doc-viewer` (PPTX) + `xlsx` (Excel/CSV) + 自定义渲染器 (PDF/图片/MD) | 每种格式用最合适的库，避免单一依赖的局限性 |
| Wiki AI Chat | 独立组件 `WikiChatPanel` | 与 WikiBrowser 解耦，便于维护 |
| 搜索预览 | 内联展开 + Markdown 渲染 | 保留外链功能，同时提供预览 |
| 状态管理 | 本地 useState | 组件间无复杂共享状态 |
| IPC 模式 | `ipcMain.handle` / `ipcRenderer.invoke` | 请求-响应模式，类型安全 |
